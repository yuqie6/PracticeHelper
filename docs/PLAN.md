# PLAN - PracticeHelper

## 总目标

做一个真正有训练感的面试训练 Agent，而不是普通八股问答壳。

---

## Phase 0 - 立项与技术栈拍板

目标：先把产品边界、技术路线和工程约束定住，避免上来乱写。

### 交付物

- README.md
- PRD.md
- PLAN.md
- ARCHITECTURE.md
- 技术栈决策

### 完成标准

- 明确 v0 / v1 边界
- 明确目标用户与使用场景
- 明确首版技术路线
- 明确前后端与 sidecar 的职责边界

---

## Phase 1 - 工程骨架

目标：起出可以持续开发的最小三端骨架。

### 预期内容

- `web/`：Vue + Vite + pnpm
- `server/`：Gin + SQLite
- `sidecar/`：FastAPI + LangGraph
- 根目录开发脚本与统一命令
- 基础 lint / format / test 配置

### 完成标准

- 三端都能本地启动
- 前端能访问 Go API
- Go 能调用 Python sidecar
- SQLite 能正常初始化

---

## Phase 2 - 画像与首页 MVP

目标：先把用户能“进来就开始用”的基础入口做出来。

### 范围

- 用户画像表单
- 用户画像保存 / 回显
- 首页空态
- 今日训练建议占位逻辑
- 最近训练记录占位逻辑
- 薄弱点 Top 5 占位逻辑

### 完成标准

- 能创建一个用户画像
- 刷新后能正确读取画像
- 首页能展示真实数据或空态

---

## Phase 3 - 项目导入 MVP

目标：把 GitHub 仓库转成项目训练可用的结构化资产。

### 范围

- GitHub 仓库 URL 输入
- 仓库抓取
- 文本白名单扫描
- chunk 切分
- 项目画像草稿生成
- 项目画像编辑与保存

### 完成标准

- 至少能成功导入 1 个公开 GitHub 仓库
- 能生成可编辑的项目画像
- 导入后的项目可用于训练

---

## Phase 4 - 八股训练 MVP

目标：做出第一条真正有用的训练闭环。

### 范围

- 主题选择
- 主问题生成
- 用户回答记录
- 评分与点评
- 至少一轮追问
- 训练复盘卡
- 弱项标签更新

### 完成标准

- 至少支持 3 个主题
- 每次训练都能形成结构化复盘
- 能记录弱项标签

---

## Phase 5 - 项目训练 MVP

目标：围绕真实项目做深挖训练。

### 范围

- 项目选择
- 基于项目画像与 repo chunks 出题
- 围绕技术选型 / 架构 / 难点追问
- 输出项目表达复盘
- 更新项目维度弱项

### 完成标准

- 至少支持录入 2 个项目
- 能围绕单项目完成一轮训练
- 复盘中能指出表达漏洞与薄弱点

---

## Phase 6 - 推荐与闭环收口

目标：让历史训练真正反馈到下一轮训练。

### 范围

- 弱项聚合
- 今日训练建议
- 推荐专项
- 最近训练记录

### 完成标准

- 首页推荐与真实弱项绑定
- 连续两次答差同一主题后排序会上升
- 弱项改善后严重度能下降

---

## 当前建议优先级

1. 统一文档口径
2. 起三端骨架
3. 接入 lint / format / test
4. 先做画像与首页
5. 再做项目导入
6. 再做八股训练
7. 最后接项目训练与推荐闭环

---

## 设计原则

- 先做有训练价值的功能，不做花活
- 先服务单用户真实需求，再考虑泛化
- 先做闭环，不先追求题库规模
- 先让反馈足够痛，再考虑包装得好看
- 用 LangGraph，但不把 v0 做成复杂多 agent 系统
- 前端风格要强识别度，但保持可读性与可维护性

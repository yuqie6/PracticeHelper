# 产品需求文档 - PracticeHelper

> 状态：本文已从最初 v0 PRD 更新为 2026-03-21 的当前产品边界快照。阶段 A / B 已完成，当前主线见 [PLAN.md](./PLAN.md) 和 [PRODUCT_UPGRADE_PLAN.md](./PRODUCT_UPGRADE_PLAN.md)。

## 1. 要解决的问题

准备后端 / AI Agent 方向面试时，核心困难不是"找不到题"，而是：

- **不知道该优先练什么** —— 缺乏对自身薄弱点的客观诊断。
- **基础知识只会背结论** —— 一被追问就暴露理解不到位，比如能说出"Redis 快是因为内存"，但说不清事件循环和数据结构的作用。
- **项目做过但讲不出来** —— 做过的项目缺少结构化表达，被问到技术选型和 trade-off 时容易卡壳。
- **练习没有连续性** —— 薄弱点不会被持续追踪，每次练完都从零开始。

现有工具（刷题平台、静态题库）解决不了这些问题，因为它们缺少：连续追问、基于真实项目的深挖、对薄弱点的长期记忆。

## 2. 目标用户

### v0 核心用户

一名准备后端 / AI Agent 方向实习的学生开发者。熟悉 Go / Python，做过若干工程项目，需要把"做过"转化为"面试能讲出来"。

### 后续潜在用户

- 校招 / 实习求职者
- 有项目经验但不擅长表达的开发者
- 想系统训练后端面试的人

## 3. 核心能力

### 3.1 用户画像

采集用户的求职背景信息，用于个性化出题和推荐：

- 目标岗位（如"Go 后端"、"AI Agent 工程师"）
- 技术栈列表
- 目标公司类型（如"大厂"、"创业公司"）
- 当前阶段（实习 / 校招）
- 目标投递时间
- 主要项目列表
- 自我感知的薄弱领域

### 3.2 基础知识训练

围绕后端核心主题进行的问答训练。当前 basics 已支持：

- Go
- Redis
- Kafka
- MySQL
- 系统设计
- 分布式
- 网络
- 微服务
- Docker & K8s

`os` 独立 topic 和跨 topic 的 mixed 出题仍属于阶段 C 待补项。

一轮训练当前是 **2-5 轮可配置** 的连续问答：
1. 用户选择主题、训练强度和轮次数
2. Agent 生成第 1 轮主问题
3. 用户回答后，Agent 评分并指出亮点和缺口；如果还没到最后一轮，就生成下一轮追问
4. 重复上一步，直到达到 `max_turns`
5. 最后一轮结束后生成复盘卡，更新薄弱点记录，并创建待复习计划

### 3.3 项目面试训练

围绕用户的真实 GitHub 项目进行深挖训练：

- 导入仓库后，系统自动分析代码并生成项目画像（摘要、亮点、难点、trade-off、可追问点）
- 用户可以编辑画像，补充系统未捕捉到的信息
- 训练时，Agent 基于项目画像和源码片段出题，重点考察：技术选型理由、架构决策、遇到的困难和解决方案、个人贡献

### 3.4 训练复盘

每次训练结束后生成一张复盘卡，包含：

- 总评（教练视角的整体判断）
- 优先修正项（`top_fix` + `top_fix_reason`）
- 回答亮点
- 主要漏洞
- 各维度评分明细
- 建议复习主题
- 下一轮训练建议

### 3.5 薄弱点记忆

系统持续记录并维护薄弱点标签，包括：

- 频繁答不好的基础知识主题
- 项目表达中反复暴露的问题
- 被追问后暴露的理解缺陷

薄弱点有 severity（严重度）和 frequency（出现频率），答差了会上升，答好了会下降；同时会随着 `last_seen_at` 逐步衰减，避免单次偶发失误长期污染推荐。首页训练建议基于“有效 severity”排序生成。

### 3.6 岗位视角训练（阶段 B 增量）

阶段 B 已经落地。现在不再是“准备接入 JD”，而是已经具备岗位视角训练闭环。

当前能力是：

- 用户可以在独立岗位页维护多份 JD 文本（手动粘贴，不做爬虫）
- 系统会把 JD 提取成结构化岗位要求，并保留分析历史
- 开始训练前，用户可以明确选择本轮参考的 JD
- basics / project 两条链路的出题、评分、追问和复盘都会引用这份 JD 的能力要求
- session / review / dashboard 都能回看当前轮次绑定的 JD

第一版不做的事：

- JD 自动抓取
- 多岗位对比推荐
- 简历联动
- 一轮训练同时绑定多份 JD

### 3.7 历史回看与复习入口

当前产品已经具备：

- 会话历史页，支持分页、模式 / topic / 状态筛选
- 首页待复习卡片，基于 `review_schedule` 展示到期内容
- 复盘卡推荐下一轮训练，训练页可以直接预填推荐参数
- 弱项趋势图和 `today_focus` / `recommended_track` 推荐闭环

## 4. 当前产品边界

### 当前已成立

- 用户画像创建与编辑
- 基础知识训练（9 个 topic）
- 项目训练（GitHub 仓库导入、画像生成与编辑）
- 多轮训练（2-5 turns）
- JD 管理、分析历史与训练绑定
- 每次训练生成复盘卡，并推荐下一轮训练
- 薄弱点持续追踪、趋势可视化和待复习入口
- 会话历史页与分页筛选

### 当前明确不做

- 在线代码执行 / 判题
- 多用户系统
- 算法训练
- 全真模拟面试长流程（30~45 分钟）
- 简历联动 / JD 自动抓取 / 多岗位对比
- 移动端
- 通用型仓库问答
- 复杂多 agent 编排

## 5. 产品形态

当前是 8 个页面。

| 页面 | 功能 |
|------|------|
| 首页 | 薄弱点 Top 5、最近训练记录、今日训练建议、待复习入口、倒计时 |
| 画像页 | 填写和编辑求职背景 |
| 岗位页 | 管理多份 JD、查看结构化要求和分析历史 |
| 项目页 | 导入 GitHub 仓库、查看和编辑项目画像 |
| 训练配置页 | 选择训练模式（基础知识 / 项目）、主题、强度、轮次数和 JD |
| 训练过程页 | 问答交互界面 |
| 复盘页 | 展示复盘卡 |
| 历史页 | 按模式 / topic / 状态筛选历史训练 |

## 6. 成功标准

当前有价值的判定条件：

- 能完成画像初始化、项目导入、basics 训练和 project 训练完整闭环
- basics / project 训练都支持多轮问答，并能在最后生成可行动的复盘卡
- 薄弱点记录准确，推荐和趋势会随有效 severity 变化而变化
- 用户可以保存多份 JD，并在训练前明确选择本轮参考的 JD
- 问题、评分、追问和复盘能明确引用该岗位真正看重的能力要求
- JD 原文变成 `stale` 后，新训练会被阻止，但旧成功快照仍可回看

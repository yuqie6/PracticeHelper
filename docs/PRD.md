# 产品需求文档 - PracticeHelper

## 1. 要解决的问题

准备后端 / AI Agent 方向面试时，核心困难不是"找不到题"，而是：

- **不知道该优先练什么** —— 缺乏对自身薄弱点的客观诊断。
- **基础知识只会背结论** —— 一被追问就暴露理解不到位，比如能说出"Redis 快是因为内存"，但说不清事件循环和数据结构的作用。
- **项目做过但讲不出来** —— 做过的项目缺少结构化表达，被问到技术选型和 trade-off 时容易卡壳。
- **练习没有连续性** —— 薄弱点不会被持续追踪，每次练完都从零开始。

现有工具（刷题平台、静态题库）解决不了这些问题，因为它们缺少：连续追问、基于真实项目的深挖、对薄弱点的长期记忆。

## 2. 目标用户

### v0 核心用户

一名准备后端 / AI Agent 方向实习的学生开发者。熟悉 Go / Python，做过若干工程项目，需要把"做过"转化为"面试能讲出来"。

### 后续潜在用户

- 校招 / 实习求职者
- 有项目经验但不擅长表达的开发者
- 想系统训练后端面试的人

## 3. 核心能力

### 3.1 用户画像

采集用户的求职背景信息，用于个性化出题和推荐：

- 目标岗位（如"Go 后端"、"AI Agent 工程师"）
- 技术栈列表
- 目标公司类型（如"大厂"、"创业公司"）
- 当前阶段（实习 / 校招）
- 目标投递时间
- 主要项目列表
- 自我感知的薄弱领域

### 3.2 基础知识训练

围绕后端核心主题进行的问答训练。v0 支持的主题：Go、Redis、Kafka，后续扩展到 MySQL、网络、操作系统、AI Agent / RAG 等。

一轮训练的流程：
1. 用户选择主题和训练强度
2. Agent 生成一个有深度的主问题（不是简单背诵题，而是需要理解和分析的问题）
3. 用户回答
4. Agent 评分并指出亮点和漏洞，同时生成一个追问（针对回答中暴露的薄弱点）
5. 用户回答追问
6. Agent 再次评分，生成复盘卡，更新薄弱点记录

### 3.3 项目面试训练

围绕用户的真实 GitHub 项目进行深挖训练：

- 导入仓库后，系统自动分析代码并生成项目画像（摘要、亮点、难点、trade-off、可追问点）
- 用户可以编辑画像，补充系统未捕捉到的信息
- 训练时，Agent 基于项目画像和源码片段出题，重点考察：技术选型理由、架构决策、遇到的困难和解决方案、个人贡献

### 3.4 训练复盘

每次训练结束后生成一张复盘卡，包含：

- 总评（教练视角的整体判断）
- 回答亮点
- 主要漏洞
- 各维度评分明细
- 建议复习主题
- 下一轮训练建议

### 3.5 薄弱点记忆

系统持续记录并维护薄弱点标签，包括：

- 频繁答不好的基础知识主题
- 项目表达中反复暴露的问题
- 被追问后暴露的理解缺陷

薄弱点有 severity（严重度）和 frequency（出现频率），答差了会上升，答好了会下降；同时会随着 `last_seen_at` 逐步衰减，避免单次偶发失误长期污染推荐。首页训练建议基于“有效 severity”排序生成。

## 4. MVP 边界

### 必做

- 用户画像创建与编辑
- 基础知识训练（至少 3 个主题：Go、Redis、Kafka）
- 项目训练（至少支持导入 2 个仓库）
- 每次训练生成复盘卡
- 薄弱点持续追踪

### 不做

- 在线代码执行 / 判题
- 多用户系统
- 算法训练
- 全真模拟面试（多轮连续、限时）
- 简历联动 / JD 解析
- 移动端
- 通用型仓库问答
- 复杂多 agent 编排

## 5. 产品形态

Web 应用，固定 6 个页面：

| 页面 | 功能 |
|------|------|
| 首页 | 薄弱点 Top 5、最近训练记录、今日训练建议、倒计时 |
| 画像页 | 填写和编辑求职背景 |
| 项目页 | 导入 GitHub 仓库、查看和编辑项目画像 |
| 训练配置页 | 选择训练模式（基础知识 / 项目）、主题、强度 |
| 训练过程页 | 问答交互界面 |
| 复盘页 | 展示复盘卡 |

## 6. 成功标准

v0 有价值的判定条件：

- 能完成 1 次画像初始化
- 能完成 1 次基础知识训练和 1 次项目训练
- 复盘卡内容具体、有针对性（不是泛化空话）
- 薄弱点记录准确，连续两次答差同一主题后 severity 会上升
- 用户主观感觉"这是在训练我，不是在陪聊"

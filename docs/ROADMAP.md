# ROADMAP - PracticeHelper

## 1. 文档目的

这份文档回答 4 个问题：

1. PracticeHelper 为什么按现在这个顺序推进
2. 哪些阶段已经真正收口
3. 当前主线到底是什么
4. 每一阶段完成后，系统应该多出什么真实能力

如果开发过程中出现分歧，默认以这份文档为推进基线。

---

## 2. 项目目标

PracticeHelper 的目标不是做一个“题库网站”，而是做一个面向后端 / Agent 求职方向的个人训练系统。

它至少要具备：

- **LLM**：负责出题、追问、评估、复盘
- **Memory**：记住用户画像、项目画像、训练记录、薄弱点
- **Tools**：导入仓库、检索项目上下文、更新弱点、创建训练 session、生成复盘

当前已经完成“训练闭环 + 记忆闭环 + 推荐闭环 + 岗位视角闭环”的基础闭环，主线不再是证明它“能不能训练一次”，而是继续提升训练深度、训练留存和长期复习价值。

---

## 3. 开发总原则

### 3.1 先做纵向闭环，不先摊平模块

正确顺序是：先让系统真的能训练一次，再让系统训练得更稳、更准，最后再让系统训练得更有针对性。

### 3.2 先让它对自己有用

判断一个功能要不要做的标准不是“像不像产品”，而是：明天会不会真的打开它练 15 分钟。

### 3.3 先把 agent 的骨头长出来

当前仍不追求多 agent 协作、长链自治、花哨 planning loop。优先级是：有记忆、有工具、有结构化决策、有真实训练价值。

### 3.4 先做规则可控，再逐步提高 LLM 自由度

先输出结构正确，再优化回答质量；先做到稳定可复现，再追求“像真人教练”。

### 3.5 不跳阶段

阶段 B 必须建立在阶段 A 已经收口的基础上。训练体验没有做稳之前，先做 JD、项目证据、学习规划，只会把问题向后转移。

---

## 4. 范围分层

### 当前已成立能力

- 用户画像
- 首页 dashboard
- 岗位 JD 管理与训练绑定
- GitHub 项目导入
- 项目画像草稿 + 手工编辑
- 基础知识训练闭环
- 项目训练闭环
- 复盘卡
- weakness memory
- 会话历史页与分页筛选
- 弱项趋势与待复习卡片
- 推荐下一轮训练

### 当前明确不做

- 算法在线判题
- 多用户 / 登录 / 权限
- 全真模拟面试长流程
- 通用 repo chat
- 向量数据库
- 复杂多 agent 编排
- 简历联动
- JD 自动抓取 / 多岗位对比

### 后续候选

- 算法训练模块
- 模拟面试模式（30~45 分钟）
- 项目证据映射
- 学习规划闭环
- 简历联动
- JD 自动抓取与聚合

---

## 5. 已完成阶段

### 阶段 A - 训练体验做稳 ✅

这一阶段的目标是把“能用”做成“好用”。当前已经完成：

- **端到端验证**
  - 真实 LLM 环境可跑通画像 → 导入项目 → basics 训练 → project 训练 → review
  - `scripts/e2e_live.py` 和 `make e2e-live` 已经能回放真实主线
  - dashboard 的 top weakness、`today_focus`、`recommended_track` 已有一致性校验

- **答题反馈 V2**
  - 训练页进度面板改成消费真实流式事件，不再是假进度
  - 反馈区改成“结论优先 + 改进建议 + 追问意图”
  - 复盘页改成动作导向布局，可直接开始推荐的下一轮训练

- **推荐质量与智能体增强**
  - 弱项记忆支持时间衰减，区分陈旧偶发问题和近期稳定弱项
  - 旧弱项再次命中时会按衰减后的热度继续累加
  - 基础题种子模板已经扩到 10 个 basics topic，`mixed` 也能按 weakness 自动跨 topic 出题
  - 追问 prompt 增加“证据不足时保守表达”约束
  - live 验证已经证明推荐会随着真实弱项变化而变化

### 阶段 A 的完成标志

- 连续使用一周不想砸掉它
- 首页建议和真实薄弱点大致一致
- 复盘不再只是终点，而是下一轮训练入口

### 阶段 B - 岗位视角接入 ✅

这一阶段的目标，是把训练从“泛化练习”推进到“围绕目标岗位练”。

当前已经完成：

- 独立 JD 页面、JD 列表和分析历史
- 默认 JD 机制，以及训练前显式选择 JD
- `training_sessions` 对 JD 与分析快照的固定绑定
- basics / project 两条训练链路对 JD 分析结果的引用
- review / session / dashboard 的岗位视角回看

### 阶段 B 的完成标志

- 用户可以维护多份 JD，并查看每份 JD 的最新分析和历史分析
- 新训练可以绑定明确的 JD 分析快照
- 问题、评分、追问和复盘都能体现岗位视角
- 旧快照可回看，新训练不会错误绑定到 stale / failed / running 状态

---

## 6. 当前主线

### 阶段 C - 训练深度与留存升级 ⬜

当前主线已经从“岗位视角接入”切到“训练深度与留存升级”。

目标不是继续给阶段 B 修边角，而是回答另外一组更现实的问题：

> 这套系统能不能让人连续练下去，而且越练越有针对性？

### 这一步要做什么

- 不再把多轮训练、历史回看、弱项趋势和岗位视角这些已完成能力当成新需求反复重做
- 把已有的 session 级复习、基础耗时审计和最小 LangGraph 编排，补成真正可追踪、可解释的能力
- 保持阶段 C 的注意力收缩在真实缺口上，而不是继续修已经收口的阶段 A / B 文档边角

### 完成标准

- P0 剩余缺口收口，不再把已做完的能力当作待开发项
- P1 半成能力进入可用、可验证、可解释状态
- 产品主线从“练一次”进一步升级到“可连续训练和持续复习”

详细方案见 [PRODUCT_UPGRADE_PLAN.md](./PRODUCT_UPGRADE_PLAN.md)。

---

## 7. 推荐执行顺序

```
阶段 A（训练体验做稳，已完成）
  → 阶段 B（接入岗位视角，已完成）
    → 阶段 C（训练深度与留存升级，当前主线）
      → 阶段 D（项目证据映射）
        → 阶段 E（学习规划闭环）
```

不要在阶段 C 还没收口前提前铺阶段 D 或 E。否则系统会在“训练深度、复习闭环和审计能力都还没稳住”的情况下，过早讨论更远的能力层。

---

## 8. 开发禁止事项

为了避免项目被做散，开发过程中默认禁止：

- 一边做岗位视角闭环一边把它扩成招聘平台 / JD 平台
- 一边做训练链路一边做多用户系统
- 为了更像 agent 而先堆多 agent 图
- 为了更像产品先做复杂视觉包装
- 为了更聪明引入向量库 / RAG / 长上下文，而不解决核心训练质量

默认优先级永远是：

**针对岗位的真实训练闭环 > 记忆闭环 > 工程稳定 > 新功能扩张**

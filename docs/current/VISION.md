# VISION - PracticeHelper

> 状态：已按 2026-03-26 当前工作区收口。
> 这份文档只回答：这个产品现在是什么、已经形成了什么价值闭环、
> 当前产品形态有哪些硬约束。
> 它不回答阶段顺序，也不回答这周具体做什么。

## 1. 一句话

PracticeHelper 现在不是“刷题站”，而是一个面向后端 / AI Agent
求职者的单用户面试训练工作台。

它已经形成的基础闭环是：

- 用户画像
- 岗位 JD 管理与分析
- 项目导入与项目画像
- basics / project 多轮训练
- 复盘卡与弱项记忆
- 下轮推荐与待复习入口

换句话说，它已经具备“诊断 -> 训练 -> 复盘 -> 记忆 -> 推荐”的主链路。

## 2. 当前核心用户

当前核心用户是一名准备后端 / AI Agent 方向实习或校招的开发者：

- 熟悉 Go / Python
- 做过若干工程项目
- 需要把“做过”变成“讲得明白、扛得住追问”

## 3. 明确不是什么

- 不是 chat-first 产品
- 不是通用 AI 求职平台
- 不是多用户 SaaS
- 不是算法刷题或代码判题工具
- 不是为了展示 agent 概念而堆复杂编排的实验场

## 4. 当前产品形态约束

当前口径先固定成下面这几条，避免一边做产品一边把主线搅散：

- 训练页仍然是主交易面，真正创造价值的交互发生在训练链路里。
- 首页负责 Diagnose / Today / Recommendation，不回退成主聊天框。
- 复盘、历史、岗位准备、记忆都围绕训练组织，不和训练平权抢入口。
- chat 只做受控辅助层，用于解释、澄清、改写、追问，不做系统主入口。
- 工作区化可以是中期演化方向，但不是当前重写领域模型的理由。

## 5. 当前已经稳定成立的工作面

当前对用户真正有意义的工作面，已经不只是一页训练页，而是围绕训练组织成了
下面这些面：

- Home：Diagnose / Today / Recommendation
- Prep：画像、岗位 JD、项目画像
- Train：basics / project 多轮训练
- Review：复盘卡、优先修正项、下一轮建议
- History：历史回看、导出、删除与遗忘
- Prompt / Audit：偏内部和高级用户的实验与审计
- 受控 chat：解释、澄清、改写、辅助追问

这里最重要的约束不是“页面有几个”，而是：

- 训练仍然是主交易面。
- 其他工作面围绕训练服务，不和训练平权争夺主入口。

## 6. 当前已经成立的事实

当前仓库已经成立的，不再是“概念上想做”：

- 用户画像、岗位 JD、项目导入三类准备面
- basics / project 多轮训练主链路
- 复盘卡、弱项记忆、待复习和推荐下一轮
- 首页 Diagnose / Today / Recommendation 承接
- Prompt 实验与审计工作面

换句话说，它已经是一个围绕训练闭环组织起来的工作台，而不是一个会出题的单页。

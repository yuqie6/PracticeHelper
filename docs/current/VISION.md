# VISION - PracticeHelper

> 状态：已按 2026-03-26 当前工作区收口。

## 1. 一句话

PracticeHelper 现在不是“刷题站”，而是一个面向后端 / AI Agent
求职者的单用户面试训练工作台。

它已经形成的基础闭环是：

- 用户画像
- 岗位 JD 管理与分析
- 项目导入与项目画像
- basics / project 多轮训练
- 复盘卡与弱项记忆
- 下轮推荐与待复习入口

换句话说，它已经具备“诊断 -> 训练 -> 复盘 -> 记忆 -> 推荐”的主链路。

## 2. 明确不是什么

- 不是 chat-first 产品
- 不是通用 AI 求职平台
- 不是多用户 SaaS
- 不是算法刷题或代码判题工具
- 不是为了展示 agent 概念而堆复杂编排的实验场

## 3. 当前产品形态约束

当前口径先固定成下面这几条，避免一边做产品一边把主线搅散：

- 训练页仍然是主交易面，真正创造价值的交互发生在训练链路里。
- 首页负责 Diagnose / Today / Recommendation，不回退成主聊天框。
- 复盘、历史、岗位准备、记忆都围绕训练组织，不和训练平权抢入口。
- chat 只做受控辅助层，用于解释、澄清、改写、追问，不做系统主入口。
- 工作区化可以是中期演化方向，但不是当前重写领域模型的理由。

## 4. 阶段判断

| 阶段 | 目标 | 当前状态 |
|:-----|:-----|:---------|
| A | 把训练体验做稳 | 已完成 |
| B | 把岗位 JD 接进训练链路 | 已完成 |
| C | 提升训练深度与留存 | 当前主线 |
| D | 补项目证据映射 | 明确后置 |
| E | 做学习规划闭环 | 明确后置 |

## 5. 当前主线判断

当前主线是阶段 C，不再是“证明系统能不能训练一次”，而是继续补这 4 类真实价值：

- 训练更深：题目、追问、复盘和上下文更贴近用户真实短板。
- 留存更强：待复习、历史回看、推荐下一轮真正驱动连续使用。
- 检索更准：第一版 RAG 已落地，但证据绑定和召回质量还要继续收口。
- 审计更清楚：prompt、评估、trace 和推荐理由能被回看和比较。

## 6. 当前不该展开的方向

- 多 agent 协作和长链自治
- chat-first 首页
- Workspace 级大重构
- 简历联动、JD 抓取、多岗位对比
- 为了“更像产品”而把已经完成的小功能重新铺一遍
